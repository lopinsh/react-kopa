# Performance & Regression Investigation Report

## 1. Performance Report

### Root causes identified
1. **Unparallelized Server Component Awaits (High Impact)**
   In [app/[locale]/page.tsx](file:///d:/WEB/ejam-kopa/app/%5Blocale%5D/page.tsx), the discovery page awaits up to 7 separate backend service calls sequentially instead of using `Promise.all()`. This creates a severe waterfall effect on the server, significantly degrading Time to First Byte (TTFB).
   *Calls affected:* [getDiscoveryCategories](file:///d:/WEB/ejam-kopa/actions/discovery-actions.ts#18-24), `getTaxonomy`, `GroupService.findCategoryBySlug`, [snapInterest](file:///d:/WEB/ejam-kopa/lib/services/discovery.service.ts#421-447), [getGroups](file:///d:/WEB/ejam-kopa/actions/discovery-actions.ts#10-17), `EventService.getDiscoverableEvents`, [getContextualTaxonomy](file:///d:/WEB/ejam-kopa/actions/discovery-actions.ts#32-38).
2. **Heavy Database Selects / Avoidable Data Transfer (High Impact)**
   `GroupService.getGroupWithContext` and `GroupService.getUserMemberships` both fetch the ENTIRE `members` array for a group just to count the non-pending members, e.g.:
   `memberCount: g.members?.filter(m => m.role !== 'PENDING').length`
   For groups with thousands of members, this fetches massive amounts of unnecessary data and causes significant memory pressure and slow parsing, especially since `getGroupWithContext` resolves 3 times per route (layout, page, generateMetadata).
3. **Repeated Context Parsing (Medium Impact)**
   While `GroupService.getGroupWithContext` strictly adheres to `React.cache()` (deduplicating the database query itself), the data mapping and array filtering (e.g., formatting members and extracting tags) executes repeatedly across [layout.tsx](file:///d:/WEB/ejam-kopa/app/%5Blocale%5D/%5Bl1Slug%5D/group/%5BgroupSlug%5D/layout.tsx), [page.tsx](file:///d:/WEB/ejam-kopa/app/%5Blocale%5D/page.tsx), and [generateMetadata](file:///d:/WEB/ejam-kopa/app/%5Blocale%5D/page.tsx#18-41).

### Quick wins
- **Parallelize Discovery Fetching:** Group the independent `await` calls in [app/[locale]/page.tsx](file:///d:/WEB/ejam-kopa/app/%5Blocale%5D/page.tsx) into `Promise.all([getDiscoveryCategories(locale), getTaxonomy(locale)])` where possible.
- **Fix the [cat](file:///d:/WEB/ejam-kopa/lib/services/group.service.ts#413-446) URL Parameter Regression (see below):** A single-line variable assignment.

### Larger investigations
- **Refactoring Group Member Counting:** The Prisma schema should be leveraged to count non-pending members directly at the database level using Prisma's `_count` combined with `where`, rather than fetching thousands of rows to `filter().length` them in JavaScript.
- **Client Component Re-renders:** A deeper pass with React DevTools is needed to ensure [GlobalSearchBar](file:///d:/WEB/ejam-kopa/components/discovery/GlobalSearchBar.tsx#32-378) and `InfiniteScrollTrigger` (which rely on scroll events and `useSearchParams`) aren't causing unnecessary re-renders of the entire discovery layout.

## 2. Regression Report

### Root cause of category filter breakage
The exact failure occurs in **[app/[locale]/page.tsx](file:///d:/WEB/ejam-kopa/app/%5Blocale%5D/page.tsx) on line 51**.
The URL generated by legacy links or external inputs uses `?category=dancing`, but the page strictly destructures [cat](file:///d:/WEB/ejam-kopa/lib/services/group.service.ts#413-446) from `searchParams`:
```typescript
const { q, city, cat, tag, view, type, limit, tab } = await searchParams;
```
Because [cat](file:///d:/WEB/ejam-kopa/lib/services/group.service.ts#413-446) is `undefined` (while `searchParams.category` is populated), the subsequent logic on **Line 71** (`if (cat) { ... }`) is bypassed entirely, causing the discovery page to load all groups without any category filtering applied.
